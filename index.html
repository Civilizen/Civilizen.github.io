SGI-STL的Allocator究竟做了什么？

发表于 2023-09-20 分类于 C++

Allocator 是什么东西？

当我们在运用STL的容器时，我们很容易地就认为那些容器帮我们管理了一切的内存管理，但事实上，我们是可以运用自己的方式管理我们容器组织内存的方式。

通俗点来说，对于STL中的容器，他们所申请需要的内存，都是交付给 Allocator 处理的。

STL中为 Allocator 定义了一套标准的接口，如果我们实现了这些接口的要求，那么就可以为容器的内存申请和释放定义一套自己的规则。

Allocator 的一个默认版本，这个版本只是对malloc的简单封装，并没有做特殊的处理，所以是被废弃的，但我们可以根据这个轮廓，修改得到我们想要的Allocator

对象是如何new出来的？

这里的第三行其实发生了两步行为：

  1. 调用 ::operator new 配置内存;
  2. 调用元素的构造函数

对应到 alloc 的行为，就是调用：

  1. alloc::allocate() 获取内存配置
  2. ::construct() 构造对象

allocator如何做的？

很容易理解, allocator 可以分为两部分分别看待：

  1. 对象的构造和析构
  2. 对象空间的配置和释放

对象的构造和析构

构造的方法很容易理解，就是直接在得到的空间上调用 placement new 后再调用对应的构造函函数。

而对于对象的析构，alloc进行了一定的优化，他会通过判断元素的类型中是否有trivial destructor。从而采取不同的优化措施。

内存的配置和释放

从学习 SGI 的内存管理出发，我们从这里看看大神究竟是怎么做的。

设计要点

  • 要向堆请求空间
  • 考虑多线程竞争（这里忽略这点，但源码是有考虑的）
  • 需要决定内存不足时的措施
  • 要优化小碎片带来的问题

双层配置器

为了考虑小区块碎片带来的问题，SGI采用了二级配置器的策略。

第一级配置器 就是在 ::operator new 上调用 C 的 malloc() 函数和free() 函数。

第二级配置器（可选择关闭） 则是采用了内存池化的思想来处理内存碎片。

  • 内存池由malloc()获取得到
  • 维护16个自由链表，其负责16种小型区块的配置能力
  • 如果需求块大于128Byte，则转调一级配置器
  • 如果内存池的空间不足，则转调一级配置器

内存配置出来以后，为了能够方便使用这些内存，我们需要一些额外的开销来保存这块内存的信息，当内存是特别小的区块时，对每一个这样的区块保存对应的信息将会浪费非常多的空间。

内存池是什么样的？

如果需求块大于128Byte，则转调一级配置器。当小于128Byte时，则交付给内存池做管理。

自由链表

16个自由链表分别掌管大小为8, 16, 24, … ,120, 128 的小额区块。

使用union的好处在于，当你以第一个字段的形式看待这个指针时，它指向的是下一个obj的地址（还未分配给客户，在自由链表中被管理着），当你以第二个字段看待时，那么该指针指向的是实际内存的地址（分配给了客户）。

每次申请区块大小时，都会向上调整到8的倍数。（申请30Byte，返回32Byte）。

我们先看看这个二级配置器类的内容有哪些？

deallocate()如何回收管理区块？

  1. 判断要求的区块有没有大于128字节，如果大于128字节就交给一级配置器来干，然后直接返回
  2. 找到想要区块对应的free list指针，判断是否有可用的区块，如果有就将区块返回，然后调整free list，返回
  3. 因为没有足够的区块可以返回，调用refill()函数填充free list
  4. 调整free list，返回结果

allocate()如何配置出区块？

这里很好理解，如果返回的区块大于128,则调用一级配置器处理。如果小于128，则找到相应的free list,采用头插法的方式将区块放回。

refill()如何填充free list?

  1. 向内存池中申请（通过chunk_alloc()函数完成）20个对应区块大小的新结点（如果内存池中不含有那么多，则可能获得的区块小于20个）
  2. 如果获取的区块为1，则将这个区块返回给客户, 然后return
  3. 将剩余的区块放入到free list中，调整free list指针。

chunk_alloc()如何从内存池中取给free list用？

  1. 计算内存池的剩余空间和申请需要的总字节数
  2. 剩余空间完全满足申请的需求，调整内存池记录的起始地址和末尾地址。然后返回空间。return
  3. 剩余空间大于等于1个区块的大小，返回剩下最多能给的区块数量给调用者，return
  4. 剩余内存空间不够给，先把剩下的内存池空间分配给合适的free list,用尽所有的内存池空间
  5. 申请一块新的内存池空间（调整start_pos），大小为2倍请求的空间 + 随着申请内存池空间次数越多越大的附加值
  6. 申请内存成功的话，返回区块给调用者
  7. 如果申请内存失败，看看free_list中还有没有比区块大小更大的free list上有空闲的区块，如果有，则把大于这个区块大小中的第一个链表中的第一个区块调整成内存池的剩余量，然后递归调用chunk_alloc()生成需要的块。
  8. 山穷水尽了，交给第一级配置器吧！！！！out-of-memory机制来点作用吧！它应该会做些什么来补救（调用new_handler看看能不能回收一些内存，不行的话就抛出异常。）

这上面最难理解的就是第七条，下面用个例子解释是如何做的。

比如现在要申请区块大小为16的区块，目前free list上还剩的区块有（24：2个，64：1个）。且无法再申请新的内存了。要从free list中找点区块来用。

那么他根据规则，会找到区块大小为24的free list，取出第一个块，把内存池的起始地址设置为这个区块大小为24的起始地址，末尾地址设置为这个24区块大小。然后重新递归调用chunk_alloc()。

递归调用chunk_alloc()的行为就会变成流程中的第三步。最后的结果就是内存池最终剩下（24-16）= 8字节的大小，返回了一个16大小的区块给客户。

我理解的C++智能指针

发表于 2023-09-17 分类于 CPP11

为什么需要智能指针？

从RAII（资源获取即初始化）编程的角度出发，使用智能指针能解决内存泄漏问题，用智能指针类封装裸指针能够保证申请的空间变量能够正常地释放，有可能作为程序员的你在很远的地方申请了资源，但是忘记了释放。

此外，它还能解决面向对象设计编程的一个问题，值传递和引用传递。某些对象在使用值传递时，会触发复制操作（return语句）。但是有些对象可能设置了不可复制的特性，那么使用值传递就不能解决我们的需求。那么传递一个指针？

又会回到上一个问题，我可能忘记释放了！！！

智能指针的区别

auto_ptr(CPP11) vs unique_ptr

首先要理解，为什么要使用auto_ptr，auto_ptr只允许有一个类实例获得了某个指针的所有权。很容易想到的一个使用场景，就是工厂模式里面的工厂。

unique_ptr 禁止了复制运算符的使用，如果要将所有权转移，则需要调用move语句。这也是一种显示的所有权转移。当然，还是慎用move语句！

shared_ptr(CPP11)

shared_ptr 允许多个类对象同时拥有一个指针实例，当对这个指针的引用技术变为0以后，就将该指针拥有的内存对象释放出来。

使用shared_ptr需要注意一些要求：

  • 不要对同一个指针多次调用构造shared_ptr，这样会导致多次释放内存。
  • 不要直接使用shared_ptr管理的那个裸指针来访问对象，因为这种方式不会让引用计数增加，可能会发生访问时内存已经释放的现象。
  • 避免出现循环引用，会导致拥有指针的双方各自等待对方释放，导致资源无法释放。

weeak_ptr

为了解决循环引用可能带来的问题，weeak_ptr应运而生，weeak_ptr不占有指针，使用它指向对象时，不会增加计数，从而可以让资源正常释放。

但是使用weak_ptr访问指针的时候，因为weeak_ptr不占有指针，我们无法像使用shared_ptr一样直接访问里面的指针对象。

我们需要调用 weak_ptr 的 lock() 方法，它可以判断对象是否还存在，如果不存在，则返回一个空的shared_ptr，如果存在则返回一个存有对象的share_ptr。

如果要使用weak_ptr构造一个shared_ptr，需要在此之前判断对象是否析构，使用的方式可以是调用ues_count()判断。

如何获取*this指针

假设你的成员函数里面需要使用该对象的智能指针，我们看看会发生什么问题？

如果参数只是需要一个裸指针，那么返回一个*this就可以解决这个问题（但还是不符合RAII），但是如果是要智能指针对象，我们可以直接调用 make_shard(*this)吗？

很明显不可以，在成员函数里，对*this指针调用了make_shared, 离开了这个成员函数的作用域，智能指针会检测到*this指针已经没有被指向了，从而触发了智能指针的析构函数。从而离开成员函数作用域以后，这个对象实例就被释放了！！！

为了解决这个问题，CPP11提供了一个模板类用于获取*this指针的shared_ptr对象。

std::enable_shared_from_this<T>

当需要在某个类的成员函数里面使用this指针，那么就可以将该类继承于这个模板类，从而在函数中调用 shared_from_this() 返回对应的this指针的智能指针。

使用 shared_from_this()函数也有一定的限制，就是不能在构造函数和析构函数中调用这个函数，因为 enable_shared_from_this 内部储存了一个 weak_ptr。shared_from_this 函数就是通过这个 weak_ptr 得到了。

这个weak_ptr在构造函数中是没有对其进行初始化的，所以在构造函数中调用shared_from_this会发生错误。同理，析构函数中也是不允许的，因为对象已经析构，weak_ptr返回空，从而没法得到对象。
